# ============================================================================
# 🔄 RATE LIMITING ARCHITECTURE (1 REQ/SEC)
# ============================================================================

PROBLEM: Free IMDB API blocks IPs that make too many requests

SOLUTION: RateLimitedQueue (core/queue.py)
   - Background thread runs _worker_loop()
   - Queue holds pending requests from users
   - Worker processes ONE request per second minimum
   - Multiple user clicks → single API call (coalescing)

FLOW:
   1. User clicks "Search" in UI
   2. cache_manager.get() called
   3. Check cache (DB/FS) → return if hit
   4. Submit to queue: cache_manager.queue.submit()
   5. Worker thread processes at 1 req/sec
   6. Result stored in queue.results dict
   7. UI can poll queue.get_result(request_id)

RETRY LOGIC (exponential backoff):
   - HTTP 429 (rate limited): wait 4s, retry
   - HTTP 503 (unavailable): wait 4s, retry
   - Timeout: wait 4s, retry
   - Max 3 retries total
   - After max retries: return error to user

THREAD SAFETY:
   - _lock for results dict (thread-safe dict operations)
   - Queue.Queue() is thread-safe by default
   - No race conditions in worker loop

# ============================================================================
# 💾 ERROR HANDLING STRATEGY
# ============================================================================

PRINCIPLE: Fail gracefully, log everything

PATTERN:
```python
try:
    # Do something
    result = risky_operation()
except SpecificError as e:
    logger.error(
        f"User-friendly message: {e}",
        exc_info=True,  # Include traceback
        extra={
            "error_code": "SPECIFIC_ERROR_CODE",
            "custom_field": "for debugging"
        }
    )
    # Handle gracefully (return default, show UI error, etc.)
    return default_value
except Exception as e:
    logger.error(f"Unexpected error: {e}", exc_info=True)
    raise  # Or return default
```

ERROR CODES:
   ✅ RATE_LIMIT_429: IMDB API rate limited
   ✅ SERVICE_UNAVAILABLE_503: IMDB API down
   ✅ REQUEST_TIMEOUT: Network timeout
   ✅ CONNECTION_ERROR: Network unreachable
   ✅ DB_CONNECTION_FAILED: MySQL unreachable
   ✅ JSON_SERIALIZATION_FAILED: Can't serialize to JSON
   ✅ JSON_PARSE_FAILED: Can't parse from JSON
   ✅ CACHE_SAVE_FAILED: Can't write to cache
   ✅ CACHE_LOAD_UNEXPECTED_ERROR: Unexpected error loading cache
   ... (many more in logger calls)

LOGGING:
   - ALL errors logged with exc_info=True (includes traceback)
   - Request ID included for tracing user requests
   - Latency in ms for performance analysis
   - Error code for categorization and filtering
   - Custom fields for context-specific debugging

# 🎯 KEY TESTING SCENARIOS
# ============================================================================

SCENARIO 1: Rate Limiting Works
   - Open app UI
   - Click "Search" 5 times rapidly
   - Observe: Only 1 request sent to IMDB API
   - Other 4 requests queued and processed at 1 req/sec
   - Total time: ~5 seconds for 5 searches

SCENARIO 2: Cache Hit
   - Search "The Godfather"
   - Wait 1 second
   - Search "The Godfather" again
   - Observe: Cache badge shows "HIT"
   - Response time: <50ms (from database)

SCENARIO 3: Cache Miss to Hit
   - Search "Inception" (new query)
   - Observe: Cache badge "MISS", latency ~2-3 seconds
   - Search "Inception" again
   - Observe: Cache badge "HIT", latency <50ms

SCENARIO 4: Error Recovery (Rate Limited)
   - If you search too fast outside the app
   - IMDB returns 429
   - Queue auto-retries with backoff
   - Eventually succeeds or shows error after 3 retries
   - Check logs/errors.json for RATE_LIMIT_429

SCENARIO 5: Database Connection Failed
   - Start app without MySQL running
   - UI shows: "Database connection failed"
   - Health check logs: DB_CONNECTION_FAILED
   - Start MySQL, refresh page
   - App auto-recovers

SCENARIO 6: Image Download Failure
   - Search returns movie with image URL
   - If image URL is broken (404)
   - Cache still saves successfully (doesn't fail)
   - Logs warning: IMAGE_DOWNLOAD_FAILED
   - Movie details still shown, just no image
